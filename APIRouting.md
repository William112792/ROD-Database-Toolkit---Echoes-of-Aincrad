# API Routing — ROD Database Toolkit

Version: 1.0.0
Generated by: `tools/build_api.py` (static resource tree) + `api/routes.js` (live REST layer)

This document specifies the toolkit's read-only API surface: a static,
file-based resource tree under `api/` and a small Express router that
serves it as REST endpoints. It is written to double as the reference
for a future Claude Skill / MCP server wrapper — every endpoint here
maps to exactly one planned "execution" function (see the Roadmap
table at the end).

## Design principles

1. **Isolated from the rest of the toolkit.** `api/routes.js` is a
   single `require()`'d module mounted with one line in `server.js`
   (`app.use("/api", require("./api/routes")(PROJECT_ROOT))`). Deleting
   `api/` removes only that line's effect — the Build Dashboard,
   Modding Guides, and the static site are unaffected. `tools/build_api.py`
   is a standalone script, not a `PIPELINE_SECTIONS` entry — it consumes
   `Content/ROD/` as a read-only downstream client, the same relationship
   the website has to that folder.
2. **Read-only.** Every route is `GET`. This is a query surface over
   already-generated data, not a mutation API (Modding Guides' own
   create/edit/delete endpoints already exist separately in `server.js`
   and are unrelated to this file).
3. **REST-standard resource shape.** Plural collection routes
   (`/api/weapons`), singular item routes (`/api/weapon/:id`),
   query-string filtering (`?category=`), and a search endpoint
   (`/api/search?q=`) — the exact shape requested when this API was
   scoped.
4. **Consistent envelope.** Every response is `{ data, meta }`. `data`
   is the resource (object or array); `meta` carries at minimum
   `{ count }` for collections. Errors are always
   `{ error: { code, message } }` with a matching HTTP status — never
   a bare string, an HTML error page, or a stack trace.
5. **Never crashes on missing data.** If `api/` hasn't been generated
   yet, every route returns `200` with an empty `data` and a `meta.warning`
   pointing at `python3 tools/build_api.py`, rather than a 500.
6. **No invented data.** Every JSON file under `api/` is a reshape of
   a file `build_pipeline.py` already produced and verified — nothing
   here computes or infers a new fact. Where the requested folder shape
   didn't map to something that genuinely exists in this export (see
   "Structs" and "Functions" below), the docs say so rather than
   fabricate it.

## Building the API tree

```bash
# 1. Build (or refresh) the main pipeline output first
python3 tools/build_pipeline.py

# 2. Then generate/refresh the static API resource tree from it
python3 tools/build_api.py
```

`build_api.py` is safe to re-run any time — it fully rewrites every
file under `api/` from the current `Content/ROD/` state.

## Static resource tree (`api/`)

```
api/
  _meta.json                  schema version, generation timestamp, per-collection counts
  items/
    weapons.json              every weapon, flattened across all weapon categories
    armor.json                every armor piece, flattened across all armor categories
    accessories.json          consumables/materials/key items, flattened
  monsters/
    monsters.json             every monster, flattened across all monster categories
    stats.json                Blueprint-sourced level/HP/curve data (Monsters > Stats)
  datatables/
    _index.json               every REAL DataTable in this export (name, row count,
                               field list, source path) — reuses the DT Inspector's own
                               catalog; no table name is invented
  structs/
    _index.json               distinct DataTable ROW FIELD SIGNATURES, grouped by which
                               tables share one — see "On structs" below
  functions/
    _index.json               Widget Blueprints with a function list — see "On functions" below
  localization/
    languages.json            the 13 supported language codes + display labels
  skills/
    active_skills.json        the 10-row Active Skills table
    sword_skills.json         Sword Skills tab data (included if built)
  tutorials/
    README.md, create_weapon.md, replace_model.md   short API-usage stubs
                               (the toolkit's REAL tutorials are guides/*.md,
                               user-authored and browsable under Modding Guides —
                               not duplicated here)
```

### On "structs"

The original scope sketched `structs/FWeaponData.json`-style files.
This UnrealPak JSON export does **not** contain standalone struct
header definitions (no `.h` / `UPROPERTY` declarations survive
export) — a DataTable's row shape is only visible as its **field
list**. `api/structs/_index.json` therefore indexes **distinct field
signatures** across every real DataTable, each given a stable id
(`RowShape_001`, …) and a list of which tables share it. This is the
closest real equivalent the data supports; inventing named structs
like `FWeaponData` that don't appear anywhere in the export would
misrepresent the source, so it wasn't done.

### On "functions"

The original scope sketched `functions/BP_WeaponManager.json`-style
files. **Widget Blueprints** (`WBP_*`) in this export do carry a real
function list (the BP Inspector already catalogs it) and are indexed
under `api/functions/`. **Gameplay Blueprints** under `Blueprints/`
(e.g. `BP_E001001`, and any `BP_WeaponManager`-style class if one
exists in a future export) are `Default__*_C` **property data** in
this export — EnemyLevel, AttackPower, and so on — not decompiled
function graphs. They carry no function list to index, so they are
intentionally **excluded** from `api/functions/` rather than listed
with an empty/misleading array. `GET /api/function/BP_WeaponManager`
returns a `404` with a message explaining this, not a silent empty
result.

## Live REST endpoints

Base URL: `http://<host>:<port>/api` (same server as the main app;
default port matches `server.js`'s `PORT`).

| Method | Path | Description |
|---|---|---|
| GET | `/api` | Discovery root — schema version, generation time, full endpoint list |
| GET | `/api/search?q=` | Substring search across items, monsters, datatables, structs, functions |
| GET | `/api/items` | All items (weapons + armor + accessories combined) |
| GET | `/api/item/:id` | Single item by numeric id or itemKey, searched across all item collections |
| GET | `/api/weapons` | All weapons; optional `?category=` (e.g. `OneHandedSword`) |
| GET | `/api/weapon/:id` | Single weapon by numeric id |
| GET | `/api/armor` | All armor; optional `?category=` |
| GET | `/api/armor/:id` | Single armor piece by numeric id |
| GET | `/api/monsters` | All monsters; optional `?category=` (e.g. `Beast`) |
| GET | `/api/monster/:id` | Single monster by titleId or titleKey, enriched with Blueprint stats when the enemy code resolves |
| GET | `/api/skills` | Active Skills + Sword Skills (if built) |
| GET | `/api/localization` | The 13 supported languages |
| GET | `/api/datatables` | Every real DataTable in the export |
| GET | `/api/datatable/:name` | Single DataTable's catalog entry by name (e.g. `DT_Eyebrow`) |
| GET | `/api/structs` | Distinct DataTable row field-signatures (see "On structs") |
| GET | `/api/struct/:id` | Single row-shape entry by `structId` |
| GET | `/api/functions` | Widget Blueprints with a function list (see "On functions") |
| GET | `/api/function/:blueprint` | Single Widget Blueprint's function list by name |
| GET | `/api/tutorials` | List of available tutorial stub names |
| GET | `/api/tutorial/:name` | A tutorial's raw Markdown content |

### Response envelope

```json
{ "data": [ /* or {} for single-resource routes */ ], "meta": { "count": 42 } }
```

### Error envelope

```json
{ "error": { "code": "NOT_FOUND", "message": "No weapon with id 9999" } }
```

Standard codes used: `NOT_FOUND` (404), `BAD_REQUEST` (400, e.g. a
missing `q` on `/api/search`). Unbuilt-data responses are `200`, not
an error — see design principle 5.

### Known limitations (stated, not hidden)

- `/api/search` is a plain case-insensitive substring match across a
  fixed set of fields per resource type — there is no ranking,
  fuzzy matching, or relevance score, because the underlying data
  doesn't support one. Results are capped at 200 and `meta.capped`
  is `true` when the cap is hit.
- `/api/monster/:id`'s Blueprint-stat enrichment only succeeds for
  monsters whose enemy code is both present and matched in
  `MonsterStats.json` (169 of 174 Blueprint-sourced monsters have a
  level curve; some monsters in the database have no Blueprint at
  all in this export) — `stats` is `null` when it doesn't resolve,
  never a guessed value.
- No pagination yet on large collections (`/api/items`, `/api/monsters`,
  `/api/datatables` all return their full set in one response) — sizes
  are small enough (hundreds, not thousands, of rows) that this hasn't
  been needed; would be a `?page=`/`?limit=` addition if a collection
  grows past what's comfortable in one payload.

## MCP / Skill roadmap

The endpoints above are the **read surface**. The following
"executions" were named when this API was scoped, for a later Claude
Skill / MCP server wrapper running in its own workspace folder. None
are implemented yet — this table records the intended mapping so a
future session can pick it up without re-deriving the design.

| Execution | Maps to | Notes |
|---|---|---|
| `search_datatable(name)` | `GET /api/datatable/:name` | direct |
| `search_asset(query)` | `GET /api/search?q=` | direct |
| `lookup_datatable(name)` | `GET /api/datatable/:name` | alias of above |
| `find_struct(id)` | `GET /api/struct/:id` | see "On structs" |
| `find_blueprint(name)` | `GET /api/function/:blueprint` | Widget BPs only |
| `list_weapon_types()` | `GET /api/weapons` grouped by `category` client-side | no dedicated endpoint yet |
| `search_localization(key)` | `GET /api/localization` + client-side lookup | full per-key resolver not yet exposed |
| `search_assets(query)` / `search_asset(query)` | `GET /api/search?q=` | same as above |
| `find_references(assetPath)` | *(not built)* | would wrap the Asset Inspector's `sidecars` maps — see `api/tutorials/replace_model.md` |
| `open_blend(file)` / `list_objects()` / `rename_object()` / `replace_material()` / `import_fbx()` / `export_fbx()` / `export_glb()` / `merge_meshes()` / `apply_modifiers()` / `recalculate_normals()` / `save()` | *(not built)* | these are Blender-side operations on downloaded sidecar files, not a data lookup — out of scope for this read API; would live in a separate execution runner with its own workspace folder, per the original request |
| `generate_datatable_row()` / `generate_lua()` / `generate_json()` / `generate_localization()` / `generate_readme()` / `generate_icon()` / `generate_thumbnail()` | *(not built)* | generation/authoring tools, not read endpoints — would be new mutation-style executions in the same future runner, deliberately kept out of this read-only API per design principle 2 |
| `validate_mod()` / `package_mod()` / `build_release_zip()` | *(not built)* | mod-authoring pipeline actions, same future runner |
| `find_duplicate_ids()` | *(not built)* | would be a cross-collection scan over `api/items/*.json` + `api/monsters/*.json`; straightforward to add as a read endpoint later (`GET /api/diagnostics/duplicate-ids`) |
| `find_mod_examples()` | *(not built)* | would index `guides/*.md` for mod-example content once enough guides exist |
| `extract_blueprint()` | *(not built)* | would wrap the existing `/api/pipeline/download-file` endpoint for a Blueprint's raw JSON — data already reachable today via `GET /api/function/:blueprint` for Widget BPs, or direct file download for gameplay BPs |

The generation/authoring/Blender-side executions are intentionally
**not** part of this read-only REST layer (design principle 2) — they
belong in a separate execution runner with its own workspace folder,
as scoped in the original request. This document will be updated when
that runner is built.
